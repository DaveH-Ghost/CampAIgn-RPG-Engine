# Version 1.4.2 Changelog (CampAIgn-RPG-Engine)

**Purpose:** Generic API so plugins can invoke other registered interaction handlers by id (skills pass/fail follow-ups and future branching plugins).

**Baseline:** **1.4.1**

**Status:** Implemented — GitHub tag **`v1.4.2`**; PyPI deferred until 1.5.0.

---

## Additions

### ``run_named_handler`` (1.4.2)

``run_named_handler(session, area, agent, obj, handler_id, params, *, source_action)`` validates params, builds a shallow ``ObjectAction`` copy from *source_action* with that handler id/params, and dispatches. ``run_interaction_handler`` uses the same path.

### ``collect_prefixed_params`` (1.4.2)

Helper for nested follow-up params stored under a branch prefix (e.g. ``pass_dest-area`` → ``dest-area``). Control keys can be skipped.

Convention for plugins: nest follow-up params as ``{branch}_{original_key}`` alongside ``{branch}_handler``.

### Handler ``param_fields`` / ``summary_template`` (1.4.2)

``register_interaction_handler(..., param_fields=[...], summary_template="...")`` stores optional **host UI hints** for action editors. Runtime dispatch ignores them. ``handler_catalog_entry(id)`` returns a JSON-friendly dict including those hints.

Field ``type`` values hosts may render: ``text``, ``textarea``, ``number``, ``select``, ``area_id``, ``coord``, ``handler_ref`` (nested handler with ``param_prefix``).

---

## Compatibility

- **``snapshot_version``** remains **5**.
- No new fields on ``ObjectAction`` — branching policy stays in plugin ``handler_params``.

---

## Studio (separate repo)

CampAIgn-RPG-Studio **1.4.2** renders ``param_fields`` in the action editor, uses ``run_named_handler`` for skills pass/fail follow-ups, and requires **`campaign-rpg-engine>=1.4.2`**.
