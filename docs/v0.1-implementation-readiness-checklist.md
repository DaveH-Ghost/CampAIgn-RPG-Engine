# Version 0.1 Implementation Readiness Checklist

**Purpose:**  
This checklist defines the key design decisions, scope, and agreements needed *before* implementing V0.1. It focuses on three main areas to keep the work contained and build directly on the validated V0 foundation (reliable single-action-per-turn structured output with full observability).

**Rule:** No code changes for V0.1 features until the relevant items on this checklist are checked off. (Use `[doc]` for checklist updates; `[code]` only when explicitly authorized for implementation.)

**Project Scope (Current Agreement):**
- Builds directly on V0: same 5x5 grid, 3 actions (move/look/speak), one action per LLM call/turn, same prompt structure, same TurnRecord/AgentTurn models, rich logging + ManualStepper for observability.
- Multiple agents can share the same world (non-interacting for V0.1).
- Runtime creation/editing/deletion of objects and agents via stepper commands (replaces the V0 special-case `sign` command).
- "Has changed" notifications are generalized to any object and apply across *all* agents that have looked at it.
- Initial world state (`create_initial_world`) remains exactly as in V0 (single "Explorer" agent + ball + sign). New agents/objects are added at runtime via commands.
- Fully manual control: human decides which agent gets a turn by typing its name in the stepper.
- Goal: Enable richer experimentation with dynamic environments and multiple independent agents while preserving the ability to observe full prompt→reasoning→action→memory/perception loops.
- Out of scope for V0.1: agent-to-agent perception/interaction, automatic turn sequencing, compound actions per turn, any GUI/visuals (see ROADMAP.md for V0.3), changes to core action schema or one-action-per-turn model, new dependencies.

See also: [ROADMAP.md](ROADMAP.md) for high-level V0.1 summary and [LONG_TERM_GOALS.md](../LONG_TERM_GOALS.md) for longer-term aspirations.

## Section 1: Generalize the "has changed" notification

- [ ] Define the generalized notification text format
  - Use dynamic object name (e.g. "[?] The Ceramic Ball has changed since you last looked at it." or similar neutral phrasing).
  - Keep the "[?]" prefix for unknown/stale objects.
  - Distinguish "never looked at" (plain "[?]") from "previously looked at but now stale" (full changed notice) if needed for clarity.

- [ ] Decide on tracking mechanism for stale/known state (per agent)
  - Leverage existing `Memory.looked_at` (current up-to-date knowledge) + `invalidate_look(object_id)`.
  - Consider adding lightweight support in Memory (e.g. a `stale` set or `ever_looked` tracking) if "never seen" vs "changed" distinction is valuable. Keep minimal.
  - `invalidate_look` and `mark_looked_at` remain the primary APIs (already general and per-agent).

- [ ] Update passive vision logic
  - Remove all hard-coded `obj.id == "obj_sign_01"` checks in `build_passive_vision`.
  - For any object not in `has_looked_at`: show "[?]" + changed notice *only if* the object was previously known to that agent.
  - The format in vision stays: `{name} ({id}), {position} - {desc}|[?]` (with optional changed text appended to the [?] case).

- [ ] Define invalidation on edit (cross-agent)
  - When an object's description (or future observable state) is edited via commands, call `invalidate_look(id)` on **every agent** in the world that currently has `has_looked_at(id)`.
  - This must happen for *all* agents (not just the "active" one).
  - `perform_look` continues to call `mark_looked_at` on success (per-agent).

- [ ] Update related areas
  - Remove or deprecate sign-specific examples in prompts/few-shots (or generalize them).
  - Update stepper `do_sign` (will be replaced in Section 3).
  - Ensure `perform_look` result strings remain consistent (use `obj.name` dynamically where appropriate).
  - Logging and observability (e.g. `state` command showing looked_at sets) should work per-agent.

- [ ] Success criteria for this section
  - Editing any object causes the appropriate agents to see the generalized changed notification on next `vision`.
  - Single-agent V0 behavior is preserved for backward compatibility.
  - No new special cases per object type.

## Section 2: Multi-agent support

- [ ] Confirm Agent and Memory models (no structural changes needed)
  - `Agent` dataclass remains the same (id, name, description, position, memory: Memory, last_action).
  - Each agent gets its own `Memory` instance (already designed this way for isolation).
  - `TurnRecord` stays per-turn (one action).

- [ ] Update World model for multiple agents
  - `agents: list[Agent]` already exists; `add_agent` is there.
  - Update `get_agent()` docs/behavior (it currently assumes "the single agent" and returns first). Keep for compat or add `get_agents() -> list[Agent]`.
  - Add helper if useful (e.g. `get_agent_by_name` or by id). Keep simple.
  - `create_initial_world()` must remain unchanged (still creates exactly one "Explorer" agent + pre-marks the sign).

- [ ] Enhance ManualStepper for multi-agent
  - The `agents` dict (name.lower() → agent) and `default(line)` dispatch for typing agent name already exist and should be the primary way to "give that agent a turn".
  - On LLM turn for a named agent: build prompt *for that specific agent*, execute `step_turn` only for it, update only its memory/state, then set it as the new "active" agent (`self.agent`).
  - `do_agents()`: enhance to list all, show active marker, positions, etc.
  - `vision`, `state`, `prompt`, manual `step`, etc. continue to operate on the current "active" agent (the last one given a turn or explicitly switched).
  - Consider adding `switch <name>` command (or make bare name typing switch without always running a full turn) for convenience.
  - When adding agents at runtime (via Section 3 commands), update `self.agents` dict and make the new one active if desired.
  - Deleting agents: remove from world + dict, handle if it was active (fall back to another or first).

- [ ] Agent turns and observability
  - `build_prompt(agent, world, ...)` and `step_turn(agent, world, ...)` are already per-agent — confirm they require no changes beyond using the correct agent instance.
  - Each agent's `looked_at` and history are private (this enables testing isolation).
  - Agents share objects/world grid but have no perception of other agents (no "other agents" in passive vision for V0.1).
  - Typing an agent's name runs its turn independently (human controls order/sequence manually).

- [ ] Success criteria for this section
  - Can have 2+ agents in the world.
  - Typing "Explorer" then "Goblin" runs separate turns for each, updating only their respective memories/positions.
  - `state` / `vision` reflect the active agent's view.
  - V0 single-agent usage (including existing tests and `get_agent()`) continues to work unchanged.

## Section 3: General editing commands (objects + agents)

- [ ] Replace the special `sign` command
  - Remove `do_sign` (and its hard-coded "obj_sign_01" + single-agent invalidate).
  - Introduce general commands in the stepper (e.g.):
    - `create-object <name> "<description>" at <x>,<y>`
    - `edit-object <id> [name "<new>"] [desc "<new>"] [pos <x>,<y>]`
    - `delete-object <id>`
    - Same pattern for agents: `create-agent <name> "<description>" at <x>,<y>`, `edit-agent <id> ...`, `delete-agent <id>`
  - Document exact syntax and examples in stepper intro/help.
  - Commands should be user-friendly for manual testing sessions.

- [ ] ID uniqueness rules
  - Objects and agents may share names.
  - IDs must be unique within their category (or globally — decide and document).
  - For new creations: implement simple unique ID generation (e.g. `obj_custom_01`, `obj_custom_02`, or based on sanitized name + counter). Scan existing to avoid collisions. (No complex ID utils module for V0.1.)
  - Editing must not allow changing an ID to a duplicate.

- [ ] Editing behavior and side effects
  - Editing an object's description must trigger generalized "has changed" invalidation for *all* agents (per Section 1).
  - Position edits for objects/agents are allowed (live move).
  - Agent edits affect name/desc/position but not private memory.
  - After any edit/create/delete: the world state, active agent's vision, etc. should immediately reflect changes (no restart needed).
  - Support editing the original sign object via general commands (to fully replace the old `sign` workflow).

- [ ] Integration with stepper and world
  - Use existing `world.add_object`, `add_agent`, `get_object_by_id`, etc.
  - After runtime adds/deletes, keep `self.agents` dict and `self.agent` (active) in sync.
  - Update `do_state`, `do_agents`, `vision` etc. to handle dynamic changes gracefully.
  - Commands should work regardless of which agent is currently active (edits are world-level).

- [ ] Backward compatibility and removal of old behavior
  - V0 single-agent + old workflows (manual step, LLM by name for the single agent, etc.) must continue to work.
  - The old `sign "text"` syntax is removed (document the migration to `edit-object obj_sign_01 desc "..."`).
  - No changes to initial world creation.

- [ ] Success criteria for this section
  - Can create/edit/delete objects and agents at runtime via clear commands.
  - Edits to objects correctly invalidate changed status for all relevant agents.
  - Unique ID enforcement works; duplicate names are allowed.
  - The stepper remains the primary (fully manual) interface; no auto-behavior added.
