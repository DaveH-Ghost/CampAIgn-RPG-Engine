## Why V0.3.2 (not multiplayer)

V0.3.1 proved the engine works through HTTP. **0.3.2** adds two missing тАЬGM at the tableтАЭ affordances:

1. **Narrator / world events** тАФ something happens in the room that is not any agentтАЩs action (thunder, a door slam, a smell). Every agent should **see** it in passive vision and **remember** it like a witnessed observation.
2. **Visual tokens** тАФ text chips work; optional `appearance` keys let realm-studio show emoji or simple icons without the sim caring about art assets.

**Multiplayer** (rooms, server-authoritative sessions, auth) is explicitly **deferred** to [LONG_TERM_GOALS.md](../LONG_TERM_GOALS.md). V0.4тАЩs multi-area work may reuse snapshot shapes later, but we are not building netcode in 0.3.2.

## Area events тАФ design (0.3.2aтАУb)

Today, observable text flows only from an **agentтАЩs** `passive_result` after a turn (`broadcast_actor_turn`). There is no **area-wide** event source.

### Target behavior

| Actor | Example |
|-------|---------|
| GM clicks **Emit eventтАж** in realm-studio | Modal: тАЬThunder rumbles overhead.тАЭ |
| Engine | `Session.emit_area_event(text)` тАФ **does not consume a turn** |
| Passive vision (all agents) | Unchanged static entity lines only (events are memory-only) |
| Memory (all agents) | Ingest as witnessed observation (environment pseudo-actor) |
| Next **Run turn** | Event appears in **Memory:** section for each agent |

### Engine API (draft)

```python
result = session.emit_area_event("Thunder rumbles overhead.")
# SessionResult(ok, message)
```

Optional CLI parity:

```text
emit-event "Thunder rumbles overhead."
```

### Snapshot (draft)

```json
{
  "recent_events": [
    { "session_turn": 3, "text": "Thunder rumbles overhead." }
  ]
}
```

Ring buffer (e.g. last **5** events) on `Area`; included in `build_area_snapshot` for UI turn log / sidebar.

### HTTP (draft)

| Method | Path | Maps to |
|--------|------|---------|
| `POST` | `/api/event` | `{ "text": "..." }` тЖТ `session.emit_area_event` |

Response: `{ ok, message, snapshot? }` тАФ same refresh pattern as `/api/command`.

### v1 rules

- Events are **room-wide** тАФ no range/occlusion in 0.3.2
- Events **persist** in the area log until evicted by buffer size (not single-turn flash)
- **No** turn increment on emit (same class as area edits)
- **No** streaming or scheduled events

## Appearance тАФ design (0.3.2c2тАУd)

Visualization-only metadata. The simulation **ignores** `appearance` for movement, perception, and prompts unless we explicitly add that later.

### Engine

| Entity | Field | Example |
|--------|-------|---------|
| `Agent` | `appearance: str` | `"tokens/explorer.png"` |
| `Object` | `appearance: str` | `"tokens/ball.png"` |

- Empty string = no custom token image (realm-studio falls back to name chip until **0.3.2d**)
- Value is a **client-resolved image path** (relative to realm-studio static assets, e.g. `/static/tokens/explorer.png`)
- Exposed in `snapshot()` on every agent/object dict
- Set via `create-agent` / `edit-agent` / `create-object` / `edit-object` CLI `appearance "..."` flag

### realm-studio (0.3.2d, not yet)

- Resolve `appearance` path to an `<img>` on the grid tile
- Create/edit modals: optional appearance path field
- Asset folder TBD (e.g. `frontend/tokens/`)
